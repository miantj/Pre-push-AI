# AI Code Review 使用指南

面向日常使用者：安装、配置、审查范围、跳过与排错。

---

## 一、安装与启用

1. 安装 VSIX → 重载 Cursor
2. 打开项目根目录（含 `.git`）
3. `Cmd+Shift+P` → **AI Code Review: 启用 AI Code Review 自动审查**

完成后 `.cursor/` 下会有：

| 路径 | 说明 |
| --- | --- |
| **`ai-code-review.说明.md`** | 推荐先看，用人话解释当前配置 |
| **`ai-code-review-last.md`** | 上次审查报告 |
| **`ai-code-review/`** | 配置、prompt、密钥等（文件夹，已 gitignore） |

文件夹内主要文件：

| 文件 | 用途 |
| --- | --- |
| `config.json` | 实际配置，编辑时有中文 Schema 提示 |
| `prompt.md` | 审查 prompt，可自定义审查标准 |
| `env` | Git hook 用的 API Key（Provider 模式） |
| `hook.sh` | hook 调用 runner（扩展自动生成，勿手改） |

> **注意**：仅安装扩展不会自动装 Git hook，必须执行「启用审查」。

---

## 二、日常使用

| 场景 | 怎么做 |
| --- | --- |
| push 前自动审 | 默认已装好 `pre-push` hook，直接 `git push` |
| **跳过审查直接 push** | `git push-skip`（见下方快捷配置） |
| 提交前先审 | 设置 `hooks` 加 `pre-commit`，或手动 **运行审查** |
| 不 push 先审 | `Cmd+Shift+P` → **运行审查** → 选范围 |
| 看审查结果 | **查看上次报告**，或打开 `.cursor/ai-code-review-last.md` |
| 改审查标准 | **打开 Prompt 文件**，或编辑 `prompt.md` |
| 恢复默认 prompt | **恢复默认 Review Prompt** |

### 审查范围（三选一）

**运行审查** 或 CLI `--scope` 时可选：

| 范围 | 含义 | 典型场景 |
| --- | --- | --- |
| **相对目标分支** (`branch`) | 本分支相对 `origin/stable` / `main` 等的累计改动 | push 前、PR 前整体检查 |
| **暂存区** (`staged`) | `git diff --cached`，即将 commit 的内容 | `pre-commit` hook、提交前快审 |
| **未提交变更** (`uncommitted`) | 暂存 + 工作区 + 新文件 | 本地改了一半想先看意见 |

各 Git hook 的默认范围：

| Hook | 默认范围 |
| --- | --- |
| `pre-push` | `branch` |
| `pre-commit` / `commit-msg` | `staged` |
| `post-merge` | `branch` |

### 扩展命令一览（Cmd+Shift+P）

| 命令 | 什么时候用 |
| --- | --- |
| **启用 AI Code Review 自动审查** | 每个项目第一次用 |
| **禁用 AI Code Review 自动审查** | 关掉自动 hook |
| **运行 AI Code Review 自动审查** | 不 push 也想先审代码 |
| **查看上次 AI Code Review 自动审查报告** | 看 FAIL 原因 |
| **设置 AI Code Review Provider API Key** | 用 DeepSeek 等 API 时 |
| **安装 AI Code Review Agent CLI 依赖** | 提示找不到 `agent` 时 |
| **打开 AI Code Review Prompt 文件** | 自定义审查规则 |
| **查看 AI Code Review 默认 Review Prompt** | 参考内置默认 prompt |
| **恢复 AI Code Review 默认 Review Prompt** | prompt 改乱了想重置 |

## 三、配置怎么改？

### 推荐：直接改 json

打开 `.cursor/ai-code-review/config.json`，鼠标悬停字段可看中文说明。

**禁用审查**：把 `"enabled": false` 即可（扩展会同步移除 hook 片段）。

**不要自动 hook、只手动审**：`"hooks": []`

**提交前也审**：`"hooks": ["pre-push", "pre-commit"]`

### 看不懂 json？

打开同目录的 **`ai-code-review.说明.md`**（启用后扩展自动生成），里面有当前配置解读、字段对照表和 Provider 配置教程。

### 进阶：Cursor 设置（团队推荐）

**设置 → 搜 `AI Code Review`**，可在**用户 settings** 里配好默认值，每个新项目「启用审查」时自动写入 `config.json` 和 `env`。

| Cursor 设置项 | 作用 | FastGPT 示例 |
| --- | --- | --- |
| `aiCodeReview.reviewMode` | Agent / Provider | `provider` |
| `aiCodeReview.providerType` | API 平台 | `codex`（兼容 OpenAI 格式） |
| `aiCodeReview.providerModel` | 模型 ID | `gpt-4o` |
| `aiCodeReview.providerBaseUrl` | API 根地址 | `https://aipre.yishouapp.com/api` |
| `aiCodeReview.providerPath` | API 路径 | `/v1/chat/completions` |
| `aiCodeReview.providerAllowCustomUrl` | 允许私有部署域名 | `true` |

**团队一次性配置（用户 settings.json）：**

```json
{
  "aiCodeReview.reviewMode": "provider",
  "aiCodeReview.providerType": "codex",
  "aiCodeReview.providerModel": "gpt-4o",
  "aiCodeReview.providerBaseUrl": "https://aipre.yishouapp.com/api",
  "aiCodeReview.providerPath": "/v1/chat/completions",
  "aiCodeReview.providerAllowCustomUrl": true
}
```

> **启用后以工作区 `config.json` 为准**；改 Cursor 设置里的 `providerAllowCustomUrl` / `reviewMode` 会自动同步 `env`，无需重新「启用审查」。  
> `baseUrl` / `path` 仅在首次启用时从 Cursor 设置写入 `config.json`；之后改 Cursor 设置不会覆盖已有 `config.json`。

## 四、配置项人话对照

| 设置 | 人话 | 一般怎么设 |
| --- | --- | --- |
| enabled | 总开关 | true |
| hooks | 何时自动审 | `["pre-push"]`；不要自动审就 `[]` |
| reviewMode | Agent 还是 API | 默认 agent |
| agent | Cursor 或 Claude | 默认 cursor |
| provider | 正在用的那一个 API 模型 | 见下文 Provider 教程 |
| baseline | 分支跟谁比 | 保持 auto |
| defaultScope | 手动审默认范围 | branch |
| timeoutMs | 单次审查最长等待 | 900000（15 分钟） |

`baseline: "auto"` 会按顺序尝试：`origin/stable` → `origin/dev` → `origin/main` → `origin/master`。找不到时先 `git fetch origin`。

## 五、Agent vs Provider

### Agent（默认）

本机 `agent` 或 `claude` 命令，跟 Cursor / Claude 订阅走。

```bash
agent --version   # Cursor Agent
claude --version  # Claude Code（config 里 agent 设为 claude）
```

### Provider

用自己的 API Key，适合 Cursor 额度不够时。**一次只配一个模型**，写在 json 的 `provider` 里即可。

#### Provider 最小配置

```json
{
  "reviewMode": "provider",
  "provider": {
    "type": "deepseek",
    "model": "deepseek-chat"
  }
}
```

`baseUrl`、`path` 留空时会按 `type` 自动用平台默认值。

#### 换平台 / 换模型

| 平台 | type | model 示例 | baseUrl（可选） | path（可选） |
| --- | --- | --- | --- | --- |
| DeepSeek | deepseek | deepseek-chat | https://api.deepseek.com | /v1/chat/completions |
| MiniMax 国际 | minimax | MiniMax-Text-01 | https://api.minimax.io | /v1/text/chatcompletion_v2 |
| MiniMax 国内 | minimax | MiniMax-Text-01 | https://api.minimaxi.com | /v1/text/chatcompletion_v2 |
| OpenAI | codex | gpt-4o | https://api.openai.com | /v1/chat/completions |
| **FastGPT 私有化** | codex | gpt-4o（应用绑定，可留默认） | https://aipre.yishouapp.com/api | /v1/chat/completions |

#### FastGPT 私有化（团队接入）

接口兼容 OpenAI 格式，实际请求：

`POST https://aipre.yishouapp.com/api/v1/chat/completions`

**工作区 config.json 示例：**

```json
{
  "enabled": true,
  "reviewMode": "provider",
  "provider": {
    "type": "codex",
    "model": "gpt-4o",
    "baseUrl": "https://aipre.yishouapp.com/api",
    "path": "/v1/chat/completions"
  }
}
```

**API Key 注意：**

| Key 类型 | 在哪获取 | 能否调对话接口 |
| --- | --- | --- |
| 账户 Key | 账户 → API Key 管理 | ❌ 会报 app key / account key 错误 |
| **应用 Key** | 应用 → 发布渠道 → API 访问 | ✅ 格式 `yishouai-xxx` |

**团队分工：**

1. 管理员在用户 settings 配好 `providerBaseUrl` / `providerPath` / `providerAllowCustomUrl`（见上文）
2. 每人各自申请**应用 Key**，`Cmd+Shift+P` → **设置 Provider API Key**（勿共享 Key）
3. 每个新项目：`Cmd+Shift+P` → **启用 AI Code Review 自动审查**

#### MiniMax 国内完整示例

```json
{
  "enabled": true,
  "reviewMode": "provider",
  "provider": {
    "type": "minimax",
    "model": "MiniMax-Text-01",
    "baseUrl": "https://api.minimaxi.com",
    "path": "/v1/text/chatcompletion_v2"
  }
}
```

#### API Key 与 env 文件

| 场景 | 做法 |
| --- | --- |
| **扩展内手动审查** | `Cmd+Shift+P` → **设置 Provider API Key**（存 SecretStorage） |
| **Git hook 自动审查** | 同上命令会自动同步到 `.cursor/ai-code-review/env`；或手动编辑 env |

> hook 进程读不到 VS Code SecretStorage，所以 Provider + hook 必须保证 `env` 文件里有 Key。

**env 文件常见内容（扩展自动生成）：**

```bash
# AI Code Review hook 环境变量（扩展自动生成，勿提交）
AI_CODE_REVIEW_ALLOW_CUSTOM_PROVIDER_URL=1
AI_CODE_REVIEW_API_KEY='yishouai-你的应用Key'
```

- `AI_CODE_REVIEW_ALLOW_CUSTOM_PROVIDER_URL=1`：使用 FastGPT 等非官方域名时必须开启；在 Cursor 设置里勾选 **允许自定义 Provider API 地址** 后会自动写入
- 含特殊字符的 key 请用单引号：`AI_CODE_REVIEW_API_KEY='your-key'`

## 六、自定义审查 Prompt

默认审查标准在 `.cursor/ai-code-review/prompt.md`。扩展首次启用时会从内置模板生成该文件。

- 编辑后保存即可生效（无需重启）
- 若 prompt 里**没有**写 `AI_CODE_REVIEW_VERDICT` 说明，运行时会**自动追加**结论行格式要求
- 结论行必须是单独一行：`AI_CODE_REVIEW_VERDICT: PASS` 或 `AI_CODE_REVIEW_VERDICT: FAIL`

内置默认标准要点：

- **只读**：不创建、不修改文件
- **只拦 critical/high**：必须有具体复现或可达攻击路径
- 风格、命名、推测性风险、旧 bug → 不 FAIL

## 七、审查报告怎么读？

报告路径：`.cursor/ai-code-review-last.md`

| 章节 | 内容 |
| --- | --- |
| 审查结论 | PASS / FAIL / 未能解析 |
| Push 提示 | 当前环境下是否会拦截 push |
| 一眼摘要 | 从 Agent 输出自动抽取影响、复现、根因、修复建议 |
| Agent 原始输出 | 完整原文，以文末 `AI_CODE_REVIEW_VERDICT:` 为准 |

hook 被拦时，终端也会打印「一眼摘要」和关键输出。

## 八、跳过与临时放行

### 完全不跑审查（推荐日常跳过）

```bash
SKIP_REVIEW=1 git push
```

#### 推荐：配置 git 别名（一次设置，长期使用）

```bash
git config --global alias.push-skip '!f(){ SKIP_REVIEW=1 git push "$@"; }; f'
```

之后：

```bash
git push-skip
git push-skip origin feat-xxx
```

> 仍会执行 husky 里**其他** pre-push 逻辑（如 rebase），只是跳过 AI 审查。

### 仍会审查，但 FAIL 时不拦截

```bash
AI_CODE_REVIEW_ALLOW_ISSUES=1 git push
```

### 其他临时放行（排错用，不推荐常态）

| 变量 | 效果 |
| --- | --- |
| `AI_CODE_REVIEW_VERDICT_LOOSE=1` | Agent 没输出结论行时放行 |
| `AI_CODE_REVIEW_SOFT_CLI=1` | Agent/Provider 进程异常时放行 |
| `AI_CODE_REVIEW_ALLOW_MISSING_CLI=1` | 找不到 `agent`/`claude` 时跳过 |

## 九、与 Husky / 现有 Hook 共存

扩展通过 **标记块** 向 hook 文件追加逻辑，不覆盖你原有脚本：

```sh
# >>> ai-code-review
…审查逻辑…
# <<< ai-code-review
```

- 自动识别 `.husky/` 与 `core.hooksPath`
- 若 hook 里已有其他逻辑，**启用时**会弹窗确认是否追加
- 配置文件损坏（JSON 非法）时，扩展会**移除**管理的 hook 片段，避免误拦

---

## 十、大 diff 与分批审查

当变更超过约 **12 万字符**（可用 `AI_CODE_REVIEW_MAX_DIFF_CHARS` 调整）时，会自动**按文件分批**审查，最后汇总 PASS/FAIL。

- 任一批 FAIL → 整体 FAIL
- 预计耗时 ≈ 批次数 × 超时时间，大改动请耐心等
- 设 `AI_CODE_REVIEW_BATCH_REVIEW=0` 可禁用分批（大 diff 可能直接失败）

---

## 十一、自动跳过的文件

以下路径不参与审查（减少噪音、节省额度）：

- `node_modules/`、`dist/`、`out/`
- `package-lock.json`、`.vsix`、`.DS_Store`

## 十二、常见问题

**push 没跑审查？**  
→ 是否 **启用审查**，`enabled` 是否为 true，hook 文件里是否有 `# >>> ai-code-review` 块。

**找不到 origin/stable？**  
→ `baseline` 保持 `auto`，然后 `git fetch origin`。

**想跳过审查直接 push？**  
→ `git push-skip` 或 `SKIP_REVIEW=1 git push`

**FAIL 但要紧急推送？**  
→ 仍会审查：`AI_CODE_REVIEW_ALLOW_ISSUES=1 git push`

**Agent 额度用尽？**  
→ 换 `agent: claude`，或改用 Provider 模式。

**Provider 请求失败？**  
→ 检查 `provider.type` / `model` / `baseUrl` / `path`，以及 `env` 里 API Key 是否有效。FastGPT 须用**应用 Key**（应用 → 发布渠道 → API 访问），不能用账户 Key。

**FastGPT 报 Key is error / app key rather than account key？**  
→ 当前 Key 是账户级 Key；到具体应用的「API 访问」重新创建应用 Key。

**私有部署 baseUrl 被忽略？**  
→ 确认 `env` 里有 `AI_CODE_REVIEW_ALLOW_CUSTOM_PROVIDER_URL=1`，或 Cursor 设置里开启 **允许自定义 Provider API 地址**。

**报告写「未能解析结论行」？**  
→ 检查 Agent 是否正常输出；若是 API 报错（如 Key 无效），先修 Provider 配置；配额/网络问题可设 `AI_CODE_REVIEW_SOFT_CLI=1` 临时放行。

**配置文件改了 hook 没同步？**  
→ 保存 `config.json` 后扩展会自动同步；也可重新执行「启用审查」。

**delete push 会审吗？**  
→ 不会，CLI 检测到删除远程分支的 push 会自动跳过。
